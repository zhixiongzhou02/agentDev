# daxiongAgent

一个用于展示 AI Agent 项目如何“先治理、后实现”的示例仓库。

它关注的不是某个模型、框架或单一功能，而是更底层的问题：

- 如何控制上下文加载，避免 token 被无意义消耗
- 如何让需求、设计、测试和实现之间保持可追溯
- 如何避免 Agent 项目随着文档增多而迅速失控
- 如何让人和 Agent 遵守同一套仓库工作流

仓库当前刻意采用“先治理、后实现”的方式启动。为了让 GitHub 首页能直接展示核心方法，本页除了入口说明，也同步展示 `docs/ai-agent-development-playbook.md` 的核心内容。

## 这个仓库适合谁

- 想把 AI Coding Agent 或内部 Agent 平台做成长期工程的人
- 想解决“文档越来越多、上下文越来越乱”的团队
- 想把 TDD、计划、设计和实现真正串起来的人
- 想找一个可以直接参考的仓库治理样例的人

- 文档地图、工作流、质量追溯和 AI 加载规则

# AI Agent 开发手册

## 目的

这份文档总结了一套实用的方法，用来开发类似 Codex 的 AI Agent 系统，并且从第一天开始就保持项目纪律。

目标不只是做出功能，还包括控制上下文成本、减少 token 浪费、保持文档可信，并确保实现能够追溯回需求和设计。

## 核心原则

### 1. 上下文是一种受管控资源

如果每个任务都把整个仓库、所有历史讨论和所有设计笔记一起塞进上下文，Agent 项目通常会很快失控。

上下文必须像预算资源一样被管理：

- 只加载当前任务所需的最小文档集合
- 把全局规则放在一个小而固定的位置
- 默认避免扫描整个文档树
- 优先使用可索引文档，而不是冗长叙述文档

### 2. 先讨论，再形成正式文档

并不是每个想法都应该立刻变成文档。

健康的工作方式应该是：

1. 先讨论问题或变更
2. 形成结论
3. 结论稳定后再创建或更新正式文档

这样可以让文档系统保持干净，避免仓库变成“半成品决策”的墓地。

### 3. 文档是受治理的资产

文档不应该被随意创建。

每份正式文档都应该有明确存在理由、固定归属位置，并且能被计划或决策记录引用。如果一份文档无法关联到计划、需求、设计或决策，它大概率就不应该存在。

### 4. 先计划，再实现

仓库应该把“直接开写代码”视为默认不允许的行为。

必须遵循的顺序是：

`plan -> requirement -> design -> test plan -> implementation -> verification`

对于小改动，其中一些步骤可以更轻，但仍然应该明确存在。

### 5. TDD 必须可追溯

测试不仅是先写测试，更重要的是说明每个测试为什么存在。

每个测试用例都应该能回溯到：

- 一个需求
- 一个设计决策
- 对应实现

如果一个测试没有需求或设计来源，它测试的很可能不是预期行为，而是偶然行为。

## 为什么这个结构有效

### `docs/00-meta`

这是治理层。

这里定义：

- 文档如何组织
- 哪些文档具有权威性
- ID 和命名如何运作
- 规划、设计、测试和实现的流程

Agent 应该优先读取这一区域。

### `docs/01-product`

这里是产品事实来源。

需求应当放在这里，而不是藏在任务描述、代码注释或聊天记录里。这样可以减少“系统应该做什么”和“代码实际上做了什么”之间的漂移。

### `docs/02-architecture`

这里存放架构和系统级设计，包括上下文控制和 token 策略。这些主题对 Agent 项目是核心内容，不应该被埋在一般工程笔记里。

### `docs/03-decisions`

决策记录能够保存关键取舍，而不需要让每份设计文档都重复讲完整个决策历史。

### `docs/04-delivery`

规划与执行信息放在这里。这样“没有计划就不能实现”就可以成为仓库规则，而不只是团队习惯。

### `docs/05-quality`

这一层把 TDD 变成一个可以审计的系统，而不是一句口号。测试策略、测试用例和追溯关系会被统一管理。

### `specs/`

这是功能级迭代的工作区。它允许团队围绕单一功能做聚焦设计和测试规划，同时避免全局文档树不断膨胀。

### `.ai/`

这个目录被刻意与产品和工程文档分开，用来存放 Agent 运行时规则：

- 先加载什么
- 默认不加载什么
- 每种工作流拥有多少上下文预算
- 任务应该路由到哪一组文档

## 文档治理规则

仓库应该强制执行以下规则。

### 规则 1. 没有计划就不应创建正式文档

只有在已有计划项或已批准决策明确需要时，才可以创建新的正式文档。

### 规则 2. 没有计划和设计上下文就不应开始实现

开始实现前，变更必须具备：

- 一份计划
- 一个需求来源
- 一份设计说明
- 一份测试计划

详细程度可以不同，但链路必须存在。

### 规则 3. 任何文档创建都必须被记录

每份正式文档都必须能从文档地图中被发现。如果一份文档无法从地图到达，它实际上就是未受管控的，应视为无效。

### 规则 4. 不允许不可追溯的测试

每个重要测试用例都应引用：

- 需求 ID
- 设计 ID
- 测试用例 ID

### 规则 5. 文档权威级别必须明确

并不是所有文档都处于同一权威级别。

一个仓库至少应定义三类级别：

- 事实来源
- 派生出的设计或执行细节
- 临时工作材料

这样 Agent 才知道该信任什么。

## 建议的 ID 体系

从一开始就使用稳定编号。

- Requirements: `FR-001`, `NFR-001`
- Module designs: `M-001`
- Decisions: `ADR-001`
- Plans: `PLAN-001`
- Tasks: `TASK-001`
- Test cases: `TC-001`

这些 ID 在计划、设计、测试以及相关提交引用中都应出现。

## 文档地图策略

仓库应有一份唯一的顶层文档地图，通常是：

`docs/00-meta/doc-map.md`

它应回答以下问题：

- 存在哪些类型的文档
- 它们位于哪里
- 谁可以创建它们
- 在什么条件下可以创建
- 它们使用什么模板
- 它们是权威文档还是派生文档
- Agent 应该在什么时候加载它们

这份文档非常关键，因为它能防止人和 Agent 把仓库当作一个毫无结构的知识垃圾堆。

## 上下文控制策略

在 Agent 项目里，最重要的优化通常不是模型选择，而是上下文纪律。

推荐默认加载顺序：

1. `.ai/loading-rules.md`
2. `docs/00-meta/doc-map.md`
3. 当前活动的 `PLAN-*`
4. 相关需求文档
5. 相关设计文档
6. 相关测试计划
7. 相关代码文件

推荐硬性规则：

- 默认绝不扫描整个 `docs/` 树
- 默认绝不加载全部测试
- 在定位到当前计划和需求来源前绝不生成代码

## TDD 与追溯策略

目标链路是：

`Requirement -> Design -> Test Plan -> Test Case -> Code`

这种方式有四个好处：

- 测试围绕预期行为建立
- 设计缺口会更早暴露
- 由于意图明确，评审会更快
- 由于行为有来源，重构会更安全

一份有用的追溯记录可以包含：

- 需求 ID
- 设计 ID
- 测试用例 ID
- 实现模块
- 验证状态

## 推荐工作流

对于每个功能或重要变更：

1. 创建一个 `PLAN-*`
2. 讨论范围与约束
3. 确认或创建需求文档
4. 编写设计说明
5. 编写测试计划和测试用例
6. 先写失败的测试
7. 实现代码
8. 让测试通过
9. 必要时更新追溯记录和文档地图

和直接写代码相比，这看起来更慢；但和事后清理一个已经失去上下文和文档控制的大型 Agent 项目相比，这会快得多。

## 初始文档集合

对于一个新仓库，应该从小开始。

第一批正式文档应当是：

- `README.md`
- `docs/00-meta/doc-map.md`
- `docs/00-meta/conventions.md`
- `docs/00-meta/workflow.md`
- `docs/01-product/vision.md`
- `docs/04-delivery/plans/PLAN-001-project-foundation.md`
- `docs/05-quality/test-strategy.md`
- `docs/05-quality/traceability.md`

这已经足够建立结构，同时避免在产品尚未形成前就把文档系统建设得过度复杂。

## 实用结论

一个强健的 AI Agent 项目，不应从模型选择或框架争论开始，而应从仓库纪律开始。

如果项目能够清楚回答以下问题，它就走在正确的轨道上：

- 需求的事实来源是什么
- 当前活动计划是什么
- 什么内容允许进入上下文
- 每个测试为什么存在
- 哪些文档具有权威性
- 是谁允许创建这份文档的

如果这些答案都很模糊，项目最终一定会为此付出代价，表现为 token 浪费、上下文噪声、行为不一致和迭代缓慢。

最简单但最有用的一条规则是：

把仓库设计成一种结构，让人和 Agent 都被迫遵守同一套有纪律的工作流。
