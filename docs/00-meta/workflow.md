# 工作流

## 目的

这份文档定义仓库变更默认遵循的交付流程。

## 默认顺序

标准顺序如下：

`plan -> requirement -> design -> test plan -> implementation -> verification`

任何步骤都不应被静默跳过。轻量工作可以压缩步骤，但整条链路必须仍然可见。

## 进入规则

开始实现前，必须先明确：

- 当前活动的 `PLAN-*`
- 需求来源
- 设计说明，必要时补建
- 验证将如何进行

## 退出规则

只有满足以下条件，变更才算完成：

- 计划项被标记为完成
- 最终行为已经被验证
- 如果变更重要，追溯链已同步更新
- 如果受管控文档发生变化，文档地图已同步更新

## 轻量变更路径

对于很小的改动，可以采用简化路径：

1. 将工作挂到一个已有的 `PLAN-*`
2. 在计划或规格里记录需求来源
3. 写出最小必要的设计说明
4. 定义验证步骤
5. 实现并验证

这条路径只是更轻，不是可以不留痕。

## 文档创建规则

只有在以下情况下，才可以创建新的受管控文档：

- 活动计划明确要求创建，或
- 已批准的决策明确要求创建

## 评审规则

评审时应重点检查：

- 是否存在活动计划
- 需求和设计链路是否可见
- 测试或验证是否映射回预期行为
- 新增文档是否受治理且可被发现
