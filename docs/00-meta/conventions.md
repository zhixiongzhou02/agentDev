# 约定

## 目的

这份文档定义整个仓库统一使用的命名、编号和状态约定。

## 命名规则

- 正式文档使用小写目录名和具有描述性的文件名。
- 计划文档使用 `PLAN-###-short-name.md`。
- 决策文档使用 `ADR-###-short-name.md`。
- 需求文档使用 `FR-###-short-name.md` 或 `NFR-###-short-name.md`。
- 测试用例使用 `TC-###-short-name.md`。
- 工作中的规格文档放在 `specs/<feature-name>/` 下，可使用普通名称。

## 编号规则

- ID 一旦创建就应保持稳定。
- 不同含义不能复用同一个 ID。
- 编号允许出现空洞。
- 相关文档正文中应显式引用对应 ID。

## 状态词汇

除非某份文档定义了更严格的状态集合，否则统一使用以下状态：

- `draft`
- `proposed`
- `active`
- `blocked`
- `done`
- `superseded`

## 文档要求

- 每份正式文档都必须在开头附近说明目的和范围。
- 每份计划都必须定义完成标准。
- 每份质量文档都必须说明验证证据如何记录。
- 每份派生文档都必须指向其来源。

## 链接规则

- 在正文中优先使用仓库相对路径。
- 引用需求、计划、决策或测试时应带上 ID。
- 避免创建无法从 `docs/00-meta/doc-map.md` 找到的孤立笔记。

## 变更纪律

- 当需求或流程发生变化时，应先更新权威文档，再进行实现。
- 当来源文档变化时，应在同一个变更中同步更新派生文档。
- 如果某份文档已经过时，应标记为 `superseded`，而不是悄悄改成别的用途。
