# FR-001 仓库启动包能力

## 状态

`active`

## 目的

定义仓库第一个真正面向实现的能力：把当前治理骨架沉淀成一个可复用的仓库启动包。

## 背景

当前仓库已经证明了一套受治理的 AI Agent 项目结构可以成立，但它仍然只是一个样例仓库。下一步需要把这套结构转化为可复用能力，使新项目能够快速获得相同的基础约束，而不是人工复制文档。

## 用户故事

- 作为项目维护者，我希望可以用一套标准启动包初始化新的 AI Agent 仓库，这样我不需要手工复制文档和目录结构。
- 作为协作者，我希望新仓库从第一天就具备计划、需求、设计、测试和 AI 上下文规则，这样团队和 Agent 可以遵守统一流程。

## 功能需求

### FR-001.1 初始化标准目录结构

系统必须能够创建最小治理骨架，包括：

- `docs/00-meta`
- `docs/01-product`
- `docs/04-delivery/plans`
- `docs/05-quality`
- `.ai`
- `specs`
- `templates`
- `scripts`

### FR-001.2 生成基础文档模板

系统必须能够在新仓库中生成基础文档，至少包括：

- `README.md`
- `docs/00-meta/doc-map.md`
- `docs/00-meta/conventions.md`
- `docs/00-meta/workflow.md`
- `docs/01-product/vision.md`
- `docs/04-delivery/plans/PLAN-001-project-foundation.md`
- `docs/05-quality/test-strategy.md`
- `docs/05-quality/traceability.md`
- `.ai/loading-rules.md`

### FR-001.3 支持项目级变量替换

系统必须至少支持以下初始化输入：

- 项目名称
- 项目简述
- 默认计划标题或项目基础标题

### FR-001.4 保留治理约束

生成结果必须保留文档地图、工作流、追溯和 AI 加载规则等治理约束，不能只生成目录而不生成约束文档。

### FR-001.5 避免静默覆盖

如果目标目录已经存在关键文件，系统不能静默覆盖，必须要求显式确认或使用明确的覆盖参数。

## 非目标

- 在第一版中支持多种技术栈模板
- 在第一版中支持复杂的交互式向导
- 在第一版中直接生成业务代码

## 验收标准

- 使用一条明确命令或一份明确流程，可以在空目录中生成受治理的仓库骨架
- 生成结果可直接通过文档地图和工作流进入下一阶段
- 新仓库中的文档包含项目级信息，而不是原仓库硬编码内容
