# 仓库启动包设计

## 状态

`active`

## 目的

定义 `FR-001` 的第一版最小实现方式。

## 设计概览

第一版采用“模板目录 + 启动脚本”的形式：

- 在 `templates/repo-foundation/` 下保存基础模板文件
- 在 `scripts/bootstrap_repo.sh` 中实现初始化流程
- 通过简单变量替换生成目标仓库内容

## 关键设计决策

### 决策 1. 先使用 Shell 脚本，而不是引入复杂运行时

原因：

- 当前仓库还没有既定语言栈
- Shell 脚本足够完成目录创建和文本替换
- 可以降低第一次实现的门槛

### 决策 2. 第一版只生成治理骨架，不生成业务代码

原因：

- 当前目标是验证仓库方法是否可复用
- 如果一开始就耦合业务代码模板，会混淆治理能力和业务能力

### 决策 3. 模板以“最小必需文档集合”为边界

第一版仅覆盖 `FR-001.2` 定义的基础文档，避免一次性扩展到完整产品文档体系。

## 拟议目录

```text
templates/
└─ repo-foundation/
   ├─ README.md.tmpl
   ├─ docs/
   │  ├─ 00-meta/
   │  ├─ 01-product/
   │  ├─ 04-delivery/plans/
   │  └─ 05-quality/
   └─ .ai/

scripts/
└─ bootstrap_repo.sh
```

## 输入参数

脚本第一版至少接收以下输入：

- `project_name`
- `project_description`
- `target_dir`

## 处理流程

1. 校验目标目录是否为空或可写
2. 创建标准目录结构
3. 从模板复制基础文档
4. 执行项目变量替换
5. 输出下一步阅读路径

## 风险与约束

- Shell 环境下的文本替换在不同平台上可能有细微差异
- 模板一旦与主仓库文档漂移，就会失去示例价值
- 如果没有覆盖保护，容易误伤已有目录

## 对测试计划的影响

测试需要覆盖：

- 空目录初始化
- 已有文件保护
- 占位符替换正确性
- 生成文档可被工作流直接消费
